kafka的一个坑6.24

发表于 2020-06-27 分类于 工作记录

-上上周的一个深坑记录，kafka集群的_comsour offset 主题，存放kafka集群元数据及偏移量的主题，默认副本竟然是1… 配置文件内这个主题的副本数和 主题副本数的配置项不是同一项，所以没法生效副本2的配置。 副本1的话，集群的意义就不存在了。单台故障，_comsour offset 分区故障集群自然要出大问题。利用json文件重写分区情况，已达到扩充副本为2的方案（如果是删减副本，也可以用），json文件可以用kafka自带脚本生成，也可以在kafka-managerweb上直接操作，也可以依照格式手写，手写结构顺序更清晰，脚本自动生成的貌似分区顺序是乱的。

工作记录esxi6.24

发表于 2020-06-27 分类于 工作记录

故障情况:

　　月初服务器单块磁盘故障，未做raid， 故障现象，不定时脱离esxi主机，掉盘，重启esxi主机又能重新连上磁盘，观察监控似乎是没做raid，导致几台虚拟机的读写，包括es的数据节点都落在了这块盘上，负载过大磁盘与esxi主机的延迟过高，达到了十几多秒，引起的掉盘。解决方案肯定是主机重做raid，这就意味着数据得被全部抹除，那么首先得把这台服务器上的虚拟机数据备份，关键是两台es数据节点，虚拟机直接迁移或者数据文件直接拷贝不太现实，几百个G得迁移一年，而且那样磁盘负载会非常高，磁盘又会掉盘， 人在机房大致想了下方案还是用es集群的分片迁移来实现，以下是那天具体方案的工作记录

故障处理

–es集群设置恢复分片策略，调低恢复带宽，调低并发恢复分片数，索引数，关闭副本恢复，仅恢复主分片
–故障节点重启上线，等待主分片缓慢恢复至es集群。观察虚拟机磁盘io情况，服务器磁盘延迟情况
–主分片恢复之后，迁移主分片至其他节点，副本分片先不去管（期间其他es节点不能再次发生故障，否则增加方案难度），这一个步骤最关键，只要把故障节点上的主分片都迁移到了其他节点上，这个故障节点就可以暂时下线了。
–分片全部迁移之后，节点下线，服务器关机，进bios，组raid
–组完raid安装系统，选用了esxi6.7u2镜像安装发现在系统安装步骤读取不到raid磁盘，回bios取消raid，再次安装测试，可以识别到每块单盘，应该是镜像驱动的问题，临时百度搜索了一下，没找到戴尔定制的镜像版本，但是发现官网esxi6.7u3版本貌似解决了很多驱动问题，尝试下载安装（在客户机房笔记本没有网络，用的手机热点，速度也不快，还吹着大空调，哎煎熬）
–重新制作一个6.7u3镜像启动盘，再次安装，发现已经可以读到raid磁盘了，本以为搞定了，结果没想到报错，gpt分区表有问题？具体报错已经不太记得请，大致是说分区表空间问题，做了个pe，连上服务器把磁盘分区全都清了一遍，再次安装，终于成功，系统安装之后简单配置下网络，重新加入vcenter
–之后新建虚拟机，安装es（可以直接从别的es节点scp目录过来，不拉数据目录就行。）更改配置文件，重新启动上线
–es集群设置 开启分片副本恢复，开启集群分片自动均衡功能，这一步有点长，不过是集群自身可以实现的功能，不需要手工介入，这时候已经从早上10点过来机房忙到晚上快7.8点了。打了车回家，，之后观察集群分片是否全部恢复，集群健康是否green，分片是否已经均衡发布。故障基本排除

反思与注意事项

–这次的教训就是服务器硬件这块一定要提早做好方案，磁盘一定要做raid，不做raid上环境，esxi的虚拟磁盘叠加的性能太差，而且根本没有磁盘冗余。
–还有一点去机房要穿外套

五一假期服务器事故

发表于 2020-05-07 分类于 工作记录

故障情况:

　　五一假期本来过得好好的，没想到3号收到系统发来的告警，线上有业务故障，同时prometheus发来告警,显示有6.7台节点宕机,幸好人就在家电脑就在身边,赶紧登陆到公司网络尝试直接ssh故障节点机器连接不上，ping节点不通，登陆vcenter webui 查看物理服务器，发现异常卡顿。这台服务器的所有虚拟机连接控制台全部黑屏，而且在vc上面对这台服务器的所有操作都异常的缓慢,这台esxi系统的服务器肯定出了些问题。但是观察这台服务器上的虚拟机节点情况，down了一台redis节点,一台mysql节点,一台ES节点，三台业务探针服务器，以及一台用于调试的和生产无关的虚拟机。redis是哨兵集群，mysql做了主从和mha，es是分布式的讲道理宕机对线上业务应该不会有太大的影响，联系公司领导和开发,最后确定是业务一个环节没做redis的故障切换重连，一直在宕机的那台redis节点报错连接超时。

故障处理

因为物理服务器故障原因较难定位，恢复速度也需要时间，线上业务并无影响，优先处理业务故障。临时解决方法,对一直在尝试连接报错redis节点的业务进程批量重启（批量重复节点任务可以
使用ansible的playbook）重新连接sentinel，建立redis连接。后续处理交给开发调试观察。
物理服务器的排查比较困难，因为之前提到的操作异常缓慢，导致点击一下虚拟机切换视图都需要
卡半分钟，这时候发现这台esxi开启了ssh端口，于是用xshell连接。连接之后发现操作也是同样卡顿，ls操作直接卡死，这时候怀疑是不是磁盘出了问题，因为一般ls操作到磁盘IO文件，之前也遇到过类似的情况，实在没办法，reboot 之后半小时没收到存活的端口。怀疑没启动成功，准备联系领导通知机房人员重启服务器，突然这台esxi又重新加入了vc,看来重启时间有点长。之后在esxiweb上发现了一个情况，那就是有块vmfs-5的虚拟磁盘存在异常情况,一会可用一会报错不可用，一会显示总量0，一会显示可用0，一会又正常，甚是诡异，在esxi的shell上面也是如此，只要涉及到这块磁盘的io操作，终端就会失去响应，实在无奈遂卸载删除，web操作还各种报错，还好在shell上强制删除成功，本来想着把磁盘里面的虚拟机数据文件拷贝出来，但是无奈IO操作卡死。于是放弃。还好这台服务器不止一块盘，之后的时间在其他的磁盘上重建了宕机的故障节点（可以在vc上的其他服务器虚拟机克隆过来可以减少很大的前期操作）es节点，redis节点，加入非常简单，mysql节点重新加入主从，mha检测各节点正常之后重启工作。

反思与注意事项

1.服务器有多块磁盘,应该做raid盘,这台近两年前的服务器在当初装系统的时候没有做raid，是很大的失误，如果做了raid-5或者raid-1的话，可能这次故障就可以避免，不过说来也稀奇，我都不记得这台服务器当初是怎么安装系统的了，为什么会没有做raid我都记不清了。甚至我都怀疑是不是我自己安装的。。。。因为后面我安装的所有esxi系统，服务器都是做了raid-5的。不过这也是一个教训和经验。
2.虚拟机克隆开机一定要先取消网卡挂载，在编辑里把网卡的√取消,开机进系统改了IP之后再勾上。切记切记，不然会影响线上正常的业务，包括一些进程的配置文件中配置的监听绑定IP之类的。切记切记！！！克隆之后的配置文件记得还原成宕机的那台的配置文件。
3.mysql虚拟机克隆之后还需要特别注意的修改auto.conf的uuid,每台节点唯一,不然主从报错。
.后续继续观察…如有变化再度更新.

Prometheus+Grafana整套监控系统部署方案(1)-安装部署prometheus

发表于 2020-03-22 更新于 2020-03-24 分类于 Prometheus

总览介绍

　　Prometheus是最初在SoundCloud上构建的开源系统监视和警报工具包 。自2012年成立以来，许多公司和组织都采用了Prometheus，该项目拥有非常活跃的开发人员和用户社区。现在，它是一个独立的开源项目，并且独立于任何公司进行维护。为了强调这一点并阐明项目的治理结构，Prometheus 在2016年加入了 Cloud Native Computing Foundation，这是继Kubernetes之后的第二个托管项目。
　　下图是摘自prometheus官方文档的prometheus整体监控系统运行框架:

拆分整体分析一下几个重要组件的功能:
1.Prometheus-prometheus采集到的监控数据以metric（指标）形式保存在时序数据库中（TSDB)并提供查询接口,数据库可存储在本地也可以存储在远程。
2.exporters-用于暴露已有的第三方服务的 metrics 以pull方式提供给 Prometheus。
3.alertmanager-从 Prometheus接受告警alerts,进行分组路由发出警告,接受者包括e-mail,webhook等
4.grafana-专业的可视化开源软件,支持导入prometheus数据源,对指标进行详细的可视化图表展示。

网络规划

　　我用两台centos7虚拟机做最小的模拟实验
192.168.88.128 prometheus+alertmanager+grafana
192.168.88.129 exporter

安装prometheus

　　首先安装,直接前往官网下载tar包安装最为快速。官方也提供docker镜像安装,如果对docker比较熟悉,可以采用docker,本文档采用二进制包安装官网下载地址 https://prometheus.io/download/,顺便把要用到的alertmanager组件和node_exporter组件也下载了



把prometheus和alertmanager的安装包上传到192.168.88.128虚拟机的/home/soft目录下,node_exporter上传到129的/home/soft目录下
解压压缩包到/home目录下
tar zxvf prometheus-2.17.0-rc.4.linux-amd64.tar.gz -C /home
更改文件夹名字为prometheus
mv /home/prometheus-2.17.0-rc.4.linux-amd64 prometheus
进入目录查看目录结构
cd /home/prometheus;ls

prometheus 可执行文件
prometheus.yml 默认配置文件
编辑默认配置文件
vim prometheus.yml

实际工作中疑难踩坑记录-kafka-oom

发表于 2020-03-21 更新于 2020-03-22 分类于 工作记录

故障情况:

　　这周的周五下午,收到监控发来的告警kafka集群出现了性能故障,其中一个节点显示宕机,登上kafka-manager的web页面看到集群broker确实是少了一个,且整体吞吐量较平时正常工作的时候下降了大概60%-70%,这样的集群肯定是存在问题的.于是登陆宕机服务器查看具体情况.通过大致时间发现了日志显示

　　果然,日志上面记载的非常详细,java.lang.OutOfMemoryError: Java heap space,经典的JVM-OOM问题,显然肯定是系统内存IO有问题。

优化方法:

　　KAFKA集群其实在搭建的时候已经测试了蛮长时间,根据当时的测试效果已经优化了部分参数的,但是这次竟然出现了问题,说明优化肯定哪里还存在的问题。首先既然是jvmOutOfMemory了,先尝试把kafka的jvmheap大小上调,因为这几台kafka配置也不高,只有四核8G,之前分配的堆大小是1G,这次上调到2G。

　　另外,这几台kafka当时吞吐量并不高,工作的时候当时对系统内存没有那么苛刻,为了性能考虑,当时禁用了swap,考虑到kafka除了要使用堆内测做java的gc及各种操作,还需要大量的堆外内存也就是系统分配给javajvm堆以外剩余的内存,因为kafka是将所有数据存储在内存中,之后flush到磁盘,堆外内存如果不够用也会影响到整个系统的稳定性从而影响系统上面的所有组件,当中也包括java的gc效率,所以综合考虑,开启了swap空间
,并做了swap的相关优化.此外查看了宿主服务器的空余内存发现,还有一点空余内存可供分配,于是给几台kafka从8G内存上调到了10G内存.

注意事项

1.kafka修改配置之后切忌不可全部同时重启所有broker,和所有分布式集群一样,需要单台重启,集群恢复之后再重启下一台,一直到所有节点重新重启过一遍后,集群恢复正常工作.
2.开启swap空间是临时选择,系统方面的优化最优选择还是直接增加虚拟机的实际内存,swap是磁盘虚拟出的内存拿来给系统做内存的冗余,类似于windows系统的虚拟内存,性能和内存相比极差,所以如果要开启必须做相关的优化,不然可能面临系统内存得到缓解,但是集群性能急剧下降的情况,这样就得不偿失负优化了.
3.所有配置做完24小时之内一直还没有出问题,不过发现swap确实已经开始使用了,虽然不多,但是说明kafka的堆外内存已经不太够用了.后续继续观察…如有变化再度更新.
